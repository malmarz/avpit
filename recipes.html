Incident Management — 3 recipes

Recipe 1.1

User Reports a Problem

When
User says something isn't working (portal, phone, email, walk-in)
Who
Service Desk (L1)
  1. Log it in BMC Helix. Every time. No exceptions.
  2. Categorize — Hardware > Laptop > Screen, etc.
  3. Prioritize — Impact × Urgency matrix
  4. Search knowledge baseiA searchable library of documented solutions and how-to guides in BMC Helix. If someone solved this problem before, the fix should be here. — known solution?
  5. Can't fix? → Escalate to L2 with full history
  6. Resolve & confirm with user
  7. Close ticket with resolution notes
Watch Out:
  • Never fix without creating a ticket first
  • Never close without confirming with user
  • New fix method? → Create a knowledge article

Output: Closed incident ticket with full history — what happened, what was done, how long it took.

Recipe 1.2

Same Problem Keeps Happening

When
Same incident type recurring — same error, same system, same pattern
Who
L2/L3 staff, Problem Manager
  1. Confirm the pattern — pull incident data from BMC
  2. Create a Problem record — link to related incidents
  3. Investigate root cause — 5 WhysiKeep asking "why?" five times. Example: Server crashed → Why? Memory full → Why? Process leaking → Why? Bug in v3.2 → Why still running? Upgrade deprioritized → Why? No one tracks vendor advisories. Root cause found., timeline, fishboneiFishbone diagram. A visual brainstorming tool that maps possible causes across categories: People, Process, Technology, Environment. Helps teams think beyond the obvious.
  4. Document workaround in KEDBiKnown Error Database. A list of problems where we know the root cause and have a temporary workaround. The service desk searches this to fix recurring issues instantly. for immediate use
  5. Submit Change Request for permanent fix
  6. Track until permanent fix is deployed
  7. Close problem after fix verified
Watch Out:
  • Don't skip the Problem record — document the root cause as organizational knowledge
  • Workaround ≠ fix. The Change Request is the real solution.

Output: Problem record with root cause, workaround in KEDB, change request for permanent fix.

Recipe 1.3

Major Incident — Critical System Down

When
Critical system down — email outage, Moodle during exams, network failure, breach
Who
Major Incident Manager + all relevant teams
  1. Declare it. If it feels major, treat it as major.
  2. Assign a Major Incident Manager — coordinates, does NOT troubleshoot
  3. Assemble the team — bridge call or Teams channel
  4. Communicate — stakeholders, immediately
  5. Restore service first — temp fix is OK
  6. Update every 30 min — even "no change" is an update
  7. Resolve & confirm
  8. PIRiPost-Incident Review. A meeting held 48-72 hours after a major incident to understand what happened, how fast we responded, and what to improve. Blameless — focuses on systems, not people. within 48-72 hours — blameless review
  9. Follow-up items — problems, changes, articles
Watch Out:
  • Major Incident Manager coordinates, doesn't diagnose
  • Communication is NOT optional
  • PIR is blameless — "how did our system allow this?" not "who did this?"

Service Request Management — 2 recipes

Recipes 2.1 & 2.2

Standard & Non-Standard Requests

2.1 Standard (in the catalog)

  1. Verify it's in the service catalog
  2. Log as Service Request (not Incident!)
  3. Check authorization — needs manager approval?
  4. Fulfill following documented procedure
  5. Notify user → Close
Requests ≠ Incidents. "I need a laptop" is a request. "My laptop is broken" is an incident. Different workflow, different SLA.

2.2 Non-Standard (not in catalog)

  1. Log it as non-standard service request
  2. Assess — needs infrastructure changes? → Change Request
  3. Estimate effort and timeline honestly
  4. Get approval — manager at minimum
  5. Fulfill or escalate
  6. Document — might this recur? Add to catalog!

Output: Completed request; potentially a new catalog item for the future.

Change Enablement — 4 recipes

Recipe 3.1

Routine Change (Standard)

When
Well-understood, low-risk, done many times — OS patch, user to group, routine firewall rule
Who
Responsible team — no CAB needed
  1. Confirm a pre-approved change modeliChange model = a documented, step-by-step procedure for a specific type of change that has been pre-approved by management. Because the risk is well-understood, no per-instance approval is needed. exists
  2. Follow the documented procedure exactly
  3. Log in BMC Helix (even pre-approved = logged)
  4. Execute the change
  5. Verify success
  6. Update CMDBiConfiguration Management Database. A live inventory of all IT assets (servers, apps, network devices) and how they connect. Update it whenever you add, change, or remove something. if config items changed
Watch Out:
  • "I've done this before" ≠ standard change. It must have a formally documented & pre-approved model.
  • Even standard changes must be logged — the change schedule needs them all.
Recipe 3.2

Planned Change (Normal)

When
Requires assessment — server upgrade, new app deploy, network change, security policy
Who
Requestor → Change Manager → CAB (if needed) → Implementation team
  1. Submit RFCiRequest for Change. A formal proposal describing what you want to change, why, what could go wrong, how to undo it if it fails, and when you plan to do it. Submitted in BMC Helix before any modification. in BMC Helix: what, why, impact, risk, rollback plan, testing, date, resources
  2. Change Manager reviews — low risk: approves directly. High risk: routes to CABiChange Advisory Board. A weekly/bi-weekly meeting where operations, security, apps, and business reps review planned changes for risks and scheduling conflicts. They advise — they don't have veto power..
  3. CAB reviews — advises approve, reject, or defer
  4. Schedule on change calendar, notify teams
  5. Execute during approved window
  6. Verify — objectives met, no negative impact
  7. Update CMDB
  8. PIR for significant changes
Watch Out:
  • Don't skip the rollback plan
  • High-risk = maintenance windows only
  • Don't stack multiple high-risk changes — if one breaks, you won't know which
Recipe 3.3

Emergency Change

When
Must act NOW — zero-day patch, active attack, fix during major outage
Who
ECAB (Emergency CAB — small subset available on short notice)
  1. Get ECABiEmergency Change Advisory Board. A small subset of CAB members reachable on short notice (phone/Teams) for urgent changes that can't wait for the next scheduled CAB meeting. approval — call/message, explain risk of NOT doing it
  2. Execute — have rollback plan, document as you go
  3. Log in BMC — retroactive OK for true emergencies
  4. Verify emergency is resolved
  5. Mandatory PIR — no exceptions
  6. Root cause — why was this an emergency? Preventable?
Watch Out:
  • Emergency ≠ shortcut for poor planning
  • "I didn't want to wait for CAB" is not an emergency
  • Too many emergencies? Your planning process is broken.
Recipe 3.4

Security Change That Affects Everyone

When
New MFA, password policy, VPN policy, device management — impacts all users
Who
Security team → Change Manager → CAB → Phased rollout
  1. Submit RFC — same process as any change
  2. Include: behavior change, comms plan, training needs, support plan, rollout phases
  3. CAB review with business stakeholders
  4. Communicate 1-2 weeks early — what, why, what to do, when, where to get help
  5. Pilot first → small group → monitor → adjust
  6. Full rollout
  7. Monitor impact — tickets, feedback, metrics
  8. PIR
Watch Out:
  • No advance communication = angry users flooding the service desk
  • Don't deploy MFA during finals week
  • The CAB exists for exactly this coordination

Supplier Management — 3 recipes

Recipe 4.1

Vendor Isn't Delivering

When
Vendor missing deadlines, poor quality, not responding
Who
Relationship owner, AVP Office (contracts)
  1. Document evidence from BMC Helix — response times, missed dates, quality issues
  2. Check the contract — penalties, escalation procedures
  3. Formal escalation — cite contract clauses + performance data
  4. Request remediation plan — how, by when
  5. Track remediation — measure improvement
  6. No improvement? → vendor leadership, penalty clauses, explore alternatives
Watch Out:
  • "They're trying their best" doesn't cut it when contract terms are violated
  • No ticket data = no leverage. This is why logging in BMC matters.
  • Check the contract BEFORE the meeting
Recipes 4.2 & 4.3

New Vendor & Contract Reviews

4.2 New Vendor Engagement

  1. Define requirements + success criteria first
  2. Contract must include: measurable SLAs, review schedule, knowledge transfer, exit planiExit plan answers: what happens when this contract ends? Covers data migration, knowledge transfer to our staff, transition support, and handover timeline. Must be agreed before the contract starts., right-to-audit, ITSM requirementiRequire vendors to log their work in our BMC Helix — not their own system or email. This gives us automatic tracking of response times, SLA compliance, and resolution quality., penalties
  3. Kickoff — vendor follows our change & incident processes
  4. Assign relationship owner
  5. Schedule monthly reviews

4.3 Vendor Contract Review

  1. Prepare scorecard: SLA compliance, quality, responsiveness, knowledge transfer, cost vs value
  2. Present data — metrics, not opinions
  3. Document action items
  4. Renewal decision:
    ✓ Performing → renew
    ⚠ Under → remediation plan
    ✗ Failing → explore alternatives

SLA & Continual Improvement — 3 recipes

Recipe 5.1

Setting Up an SLA

When
Defining measurable targets for a service
Who
Service owner + customer representative
  1. Identify the service
  2. Ask the customer — what do they need?
  3. Define targets: availability, response time, resolution time by priority
  4. Verify measurable — can BMC Helix track this?
  5. Check supporting chain — OLAsiOperational Level Agreement. Internal commitments between your own teams. Example: "Network team diagnoses network issues within 1 hour." Not a legal contract — an internal promise that supports your SLA. + vendor UCsiUnderpinning Contract. The legally binding contract with your external vendor that must support your SLA. If your SLA promises 4 hours but the vendor contract allows 8 hours, you have a gap. must be faster than your SLAiService Level Agreement. Your measurable promise to the customer about service quality — how fast you respond, how quickly you fix things, how available systems will be.
  6. Document in plain language
  7. Set up monitoring — dashboards in BMC
  8. Review monthly
Watch Out:
  • Don't set targets you can't meet — destroys credibility
  • Include at least one experience metric (satisfaction)
  • SLAs without monitoring are just documents
Recipes 5.2 & 5.3

SLA Breach & Continual Improvement

5.2 SLA Breach — What to Do

  1. Acknowledge — don't hide. Transparency builds trust.
  2. Analyze cause — capacity? process? vendor? one-off?
  3. Communicate to customer — what happened, what you're doing
  4. Contractual consequences if applicable
  5. Improvement action → add to CIRiContinual Improvement Register. A prioritized list of improvement ideas with owners, targets, and timelines. Not a wish list — each item is tracked to completion.
  6. Track next period to confirm resolved

5.3 Starting Continual Improvement

  1. Baseline — measure current "before" number
  2. Pick ONE thing to improve
  3. Set target — specific: "VPN MTTRiMean Time to Resolve. Average time from when an issue is reported to when it's fully fixed. Example: 10 VPN incidents took 40 total hours → MTTR = 4 hours. from 4h → 2h in 3 months"
  4. Design the improvement
  5. Execute — most orgs skip this step
  6. Measure — did it improve?
  7. Standardize or adjust
"We feel like things are better" is not improvement. Numbers are improvement.
Recipes 6.1 & 6.2

Knowledge & Documentation

6.1 Creating a Knowledge Article

  1. Check — article already exists? Update it.
  2. Write in plain language — L1 must understand
  3. Structure: Title, Symptoms, Cause, Resolution, Applies To, Last Verified
  4. Review → Publish → Link to original ticket
Test: could a new L1 agent follow this? If not, rewrite it.

6.2 When Someone Leaves or Transfers

  1. Identify what they know that isn't documented
  2. Knowledge capture sessions — how-tos, troubleshooting, vendor contacts, workarounds
  3. Pair with replacement — 1-2 weeks shadowing
  4. Verify docs — replacement follows guides without help
  5. Update ownership — CIs, services, processes reassigned
Recipes 7.1 & 7.2

Budgeting & Cost Management

7.1 Service Cost Estimate

  1. Direct costs — hardware, licenses, staff dedicated to this service
  2. Indirect costs — network, storage, overhead (allocate by usage or headcount)
  3. Full lifecycle — setup, maintenance, operation, retirement
  4. Calculate cost per user or per transaction
  5. Document assumptions

7.2 Justifying IT Investment

  1. Business terms — not "50K for servers" but "enables online exams for 5,000 students"
  2. TCOiTotal Cost of Ownership. The full lifecycle cost: purchase + implementation + annual maintenance + staff time + eventual retirement/migration. Prevents comparing just sticker prices. over 3-5 years
  3. Compare alternatives — at least two options
  4. Cost of doing nothing
  5. Run/Grow/TransformiRun = keeping lights on (60-80%). Grow = enhancing capabilities (15-25%). Transform = building new things (5-15%). If 90% goes to Run, there's no room for innovation. framing
Note: BMC Helix is not a budgeting tool. It provides data inputs (ticket volumes, resource utilization, vendor SLA metrics, asset inventory) but actual budgeting and TCO analysis is done in spreadsheets or your ERP/finance system.
