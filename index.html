Part 1

What is ITIL?

Best practices for managing IT services so that work is visible, measurable, and improvable — instead of invisible, unmeasurable, and stuck in email.

ITIL gives you

  • A common language — everyone means the same thing
  • Clear practices — how to handle each type of IT work
  • Measurement — know if things are getting better or worse
  • Improvement — a built-in mechanism for getting better

ITIL is NOT

  • Not a product — BMC Helix is a tool, ITIL is the thinking
  • Not bureaucracy — done well, it speeds things up
  • Not rigid — you adopt what you need
  • Not org certification — only individuals certify
Evolution

The journey to ITIL 4

No ITIL (where we are)

  • Work lives in email threads
  • No common vocabulary
  • No measurement or SLAs
  • Knowledge in people's heads
  • Fire-fighting, no prevention
  • No visibility into workload

ITIL v3 (2007–2019)

  • Rigid processes you must follow
  • IT delivers TO the business
  • Control and restrict change
  • Sequential lifecycle stages
  • Heavy documentation focus
  • 26 formal processes

ITIL 4 (2019–present)

  • Flexible practices you adapt
  • Co-creates value WITH business iInstead of IT building a system and handing it over, IT and the faculty work together to define what's needed, test it, and improve it. Example: Rather than IT deciding the Moodle exam setup alone, instructors and IT co-design the exam workflow together.
  • Enable and facilitate change
  • Interconnected value chain
  • Practical, adopt what you need
  • 34 flexible practices
We're jumping from "No ITIL" to ITIL 4. That's actually an advantage — no v3 baggage. We adopt the modern, flexible version from day one.
Guiding Principles

7 principles — every decision passes through these

1. Focus on Value

If an activity doesn't create value, question it.

2. Start Where You Are

Assess what works. Keep it. Improve it.

3. Progress Iteratively

Small steps, measure, adjust, repeat. Critical for us.

4. Promote Visibility

No hidden queues. No shadow processes. Critical for us.

5. Think Holistically

No team operates in isolation.

6. Keep It Simple

Minimum steps. Eliminate waste.

7. Optimize & Automate

Fix the process first, THEN automate it.

Principles 3 & 4 are our most critical — we can't transform overnight and we can't keep work invisible in email.

9:15 Technician sees it. Emails back: "Try restarting."
10:00 Still broken. Forwarded to network team. They ask Dr. Ahmad to describe the problem again.
11:30 Network engineer fixes firewall rule. Emails back through chain.
??:?? Department head calls AVP's office. No one knows the status.
End of day — nothing recorded: how long it took, root cause, whether it happened before, whether it will happen again. This happened with the MFA incident — 13 days, no ticket.
World 2

World 2: How ITIL changes this

8:30 Dr. Ahmad opens portal. Selects "VPN Issue." System creates Incident #4521, P3 priority.
8:32 Agent searches knowledge base → finds solution → sends fix instructions.
8:45 Dr. Ahmad follows instructions. It works. Ticket closed. Total: 15 minutes.
Auto System detects 5th VPN incident this month → flags a Problem.
Auto Root cause found → Change Request submitted → patching process fixed permanently.
End of day: Logged, tracked, resolved in 15 min. Pattern detected. Root cause found. Permanent fix submitted. Dashboard updated.

Not a different technology.
A different way of working.

BMC Helix is the tool. ITIL is the thinking. What actually changes is behavior.

Service Request Management

Incident = "something broke"
Service Request = "I need something"

Both are tickets but follow different workflows and different SLAs.

Common examples:

  • New laptop / hardware
  • VPN access setup
  • Software installation
  • Password reset
  • Folder/permission access

The Service Catalog

The "menu" of services users can request. Published on the self-service portal:

  • Description of the service
  • Who can request it
  • Approval workflow
  • Expected fulfillment time
  • Who fulfills it
Transforms IT from a black box into a transparent service provider.

Change Enablement

Renamed from "Change Management" → Change Enablement

Think brakes on a car — not there to stop you, but to let you go fast safely.

TypeRiskApprovalExample
StandardLow, repeatablePre-approvedPassword reset, routine patch
NormalModerate–High RFC iRFC = Request for Change. A formal proposal document that describes what you want to change, why, what could go wrong, and how to undo it. It's submitted before any modification to IT systems. + assessment Server upgrade, new app deploy
EmergencyCriticalFast-track (ECAB)Zero-day patch, outage fix
CAB (Change Advisory Board) iA regular meeting (weekly/bi-weekly) where representatives from operations, security, applications, and business review planned changes. They advise on risks and timing — they don't have veto power. Think of it as a coordination roundtable. — reviews planned changes for risk and coordination.
ECAB (Emergency CAB) iA small subset of CAB members available on short notice (phone/Teams) to approve urgent changes that can't wait for the next scheduled CAB meeting. Used for active security threats or major outages. Key difference from CAB: speed over thoroughness, but must still document and do a post-review. — a fast-track version for urgent changes that can't wait.
Part 4 — Security Coordination

Not stepping on each other's feet

The Traditional Problem

  • Security says "no" without business context
  • Blanket controls that frustrate operations
  • Separate processes, no shared visibility
  • Security seen as adversaries
→

ITIL's Solution

  • Security submits RFCs like everyone else
  • Security officer sits on the CAB
  • Shared change schedule — no surprises
  • Same process, shared visibility
The core insight: Security changes go through the same change process — risk-assessed, reviewed by the CAB, communicated in advance, phased where possible, reviewed after implementation.

Service Level Management

An SLA iService Level Agreement. A written agreement between IT and its customers that defines measurable targets for service quality — how fast we respond, how quickly we fix things, and how available systems should be. = a measurable promise about service quality.

PriorityResponseResolutionSupport
P1 Critical15 minutes2–4 hours24/7
P2 High1 hour8–24 hoursBusiness + on-call
P3 Medium4 hours1–3 daysBusiness hours
P4 Low1–2 days5–10 daysBusiness hours
We currently have no SLAs. Requests go to email and "get done when they get done." SLAs make "on time" measurable and teams accountable.
How SLAs connect

Your SLA promise is only as strong as its weakest link

When you promise a dean "email issues fixed in 4 hours," three layers must support that promise:

SLA iService Level Agreement. The promise you make to your customer (faculties, deans, staff). Example: "P1 email incidents resolved within 4 hours."

Your promise to the customer.
"We will resolve P1 email incidents within 4 hours."

Between: AVPIT ↔ KU Faculties

OLA iOperational Level Agreement. Internal targets between your own teams. Not a legal contract — an internal commitment. Example: "Network team will diagnose network-related P1 issues within 1 hour."

Internal team commitments.
"Network team diagnoses within 1 hour."

Between: AVPIT teams ↔ each other

UC iUnderpinning Contract. The legally binding contract with your external vendor. If your SLA promises 4 hours but your vendor contract allows 8 hours, you have a gap that will cause SLA breaches.

Vendor contractual obligations.
"Microsoft responds to O365 incidents within 2 hours."

Between: AVPIT ↔ BMC, Diyar, Microsoft…

The alignment rule: If your SLA promises 4 hours, your OLAs and UCs must each be faster than 4 hours — otherwise you will breach.
Watermelon SLAs: When metrics say "all green" but users are unhappy. Fix: add experience surveys alongside operational metrics.
Part 5 — Accountability

Supplier Management

CategoryMgmt LevelOur Examples
StrategicExecutive partnershipBMC, Microsoft, Google
TacticalRegular reviews, milestonesDiyar (SI partner)
OperationalStandard contractsHardware vendors, cabling
CommodityMinimal, price-drivenOffice supplies

How to hold vendors accountable:

  1. Measurable targets in contracts — not "best efforts" but "P1 response within 30 min"
  2. Regular performance reviews — monthly, documented, with scorecards
  3. Require use of our ITSM iWhen vendors log their work in BMC Helix instead of reporting via email, you get automatic tracking of their response times, resolution quality, and SLA compliance. No more "he said/she said" — the data speaks. — vendor work logged in BMC Helix, not email
  4. Knowledge transfer obligations — contractual, not aspirational
  5. Exit planning from day one iExit planning answers: "What happens when this vendor contract ends?" It includes: how data is migrated, how knowledge is transferred to our team, how we transition to a replacement, and how long the vendor must support us during handover. Example: We've used Diyar for BMC Helix for 3 years — if we switch to ServiceNow, who knows how our system is configured? Exit planning prevents that crisis.

The Run / Grow / Transform Lens

  • Run (60–80%) — keeping lights on
  • Grow (15–25%) — enhancing capabilities
  • Transform (5–15%) — building new things

If 90% goes to "run," there's no room for innovation.

Maturity

Where we are, where we're going

LevelNameWhat it meansStatus
0 None iNo process exists. Work is completely ad-hoc and chaotic. Success depends on individual heroics. Example at KU: No formal incident logging — issues handled via email, WhatsApp, or walk-ins with zero tracking. Ad-hoc, chaotic, no process ← Some practices here
1 Initial iProcess is recognized but unstructured. Some people follow a pattern, but it's not documented or consistent. Example at KU: We know we should log tickets in BMC, but it's inconsistent — some do, most don't. Recognized but inconsistent ← Some here
2 Repeatable iBasic processes established. Similar tasks handled the same way most of the time. Some documentation exists. Example: All incidents are logged in BMC, categorized consistently, and assigned to the right team. Consistent, basic documentation ← Target: 6 months
3 Defined iProcesses are documented, standardized, and trained. Consistent outcomes regardless of who does the work. Example: Written SLAs, documented workflows, new staff trained on processes, service catalog published. Documented, trained, consistent ← Target: 12–18 months
4 Managed iProcesses measured and controlled with KPIs. Data-driven management decisions. Performance is predictable. Example: Monthly SLA compliance reports, trend analysis, capacity planning based on data. Measured, data-driven decisions Future goal
5 Optimizing iContinuous improvement embedded in culture. Innovation and optimization are routine. Processes constantly refined. Example: Proactive problem management, automated workflows, AI-assisted ticket routing. Continuous improvement culture Long-term aspiration
We don't need Level 5 everywhere. Level 3 for most practices is a dramatic improvement and a realistic 12–18 month target.
